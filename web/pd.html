GPU EXPLAINER · PD 分离

Prefill 和 Decode，
拆到两个 GPU 池各干各的

上一页（SGLang 引擎）讲的是一个引擎实例内部怎么调度。但 prefill（算力重）和 decode（访存重）是两种完全不同的工作负载，塞在同一组 GPU 上永远互相干扰。PD 分离（Prefill/Decode Disaggregation）是更狠的解法：把两个阶段拆到不同的 GPU 池，各自挑最优的并行配置和 batch 策略，中间用 RDMA 高速搬运 KV cache。本页先讲清"为什么分、分了之后长什么样"，再拆 SGLang 的具体实现。

01Prefill 和 Decode 是两种生物

一次推理 = 一次 Prefill（把整段 prompt 一次算完、灌 KV cache）+ 成百上千步 Decode（每步只算 1 个新 token）。同一个模型、同一段权重，这两个阶段在 GPU 上的行为却几乎相反：

一句话记住这对矛盾 Prefill 希望这台机器是算力怪兽，decode 希望这台机器是带宽怪兽 + 超大 batch。合并部署时你只能买一种机器、选一套并行配置——永远在折中。这正是 真实数据流页实测里 SM 活跃度和 HBM 活跃度在 prefill/decode 阶段跷跷板式变化的原因。

还有一个更隐蔽的问题：DP Attention 的步调失衡

上一页讲过 MLA 模型（DeepSeek 系）要开 DPA（按请求切 attention）。DPA 的各个 rank 每步 forward 必须同步——可合并部署下，有的 rank 这一步在跑重型 prefill、有的在跑轻量 decode，快的 rank 只能空转等最慢的。整组卡的节奏被最重的那个 rank 绑架。PD 分离后每个池里的负载同质（P 池全是 prefill、D 池全是 decode），同步不再有快等慢的浪费。这两个问题（prefill 打断 + DPA 失衡）正是 SGLang 官方文档给出的做 PD 分离的两大动机。

03PD 分离：空间上拆开，KV 走网络 可交互

思路一句话：chunked prefill 是在"时间维度"切碎干扰，PD 分离是在"空间维度"消灭干扰。Prefill 池只做 prefill、算完 KV 立刻卸载（生命短）；Decode 池只做 decode、长期持有 KV、把 batch 堆大。两池之间用 RDMA 搬 KV cache。按 ▶ 看一个请求怎么流过整套系统：

关键观察：第一个 token 在 P 池产出，其余全在 D 池 Prefill 算完整段 KV 的同时就产出了第 1 个 token（这决定 TTFT）；token 2, 3, 4… 全部由 D 池产出（这决定 TPOT）。中间那次 KV 传输横在 TTFT 的关键路径上——传得不够快，整个架构就白搭。这是下一节的主题。

业界都在这么干

DeepSeek 自家跑 V3/R1 就是 PD 分离：prefill 集群大 TP 吃算力（GPU 利用率 ~80%），decode 集群大 EP + 大 batch 吃带宽，比合并部署吞吐高 30–50%。Mooncake（Moonshot/Kimi 开源，后来成了 SGLang 默认传输后端）更进一步：P 不直传 D，而是写进分布式 KV 池（DRAM+NVMe 多层），任何 D 节点都能读——同一个 system prompt 的 KV 全集群算一次就够。NVIDIA Dynamo、vLLM、TensorRT-LLM 也都有各自的 disaggregation 方案。SGLang 曾在 96 张 H100 上用 PD 分离 + 大规模 EP 复现 DeepSeek 官方吞吐。

04代价：KV cache 要搬家，而且要搬得飞快 可交互

PD 分离唯一的新增成本 = P → D 的 KV 传输。它必须塞进 TTFT 预算里（通常留 ~200ms），所以带宽是硬门槛。拖一拖滑块感受一下：prompt 越长、模型 KV 越胖、并发越高，对网络的要求呈倍数上涨。

竖线 = 200ms 预算。TCP 走 CPU 内存中转直接慢一个数量级，所以 PD 分离基本等于强制要求 RDMA + GPU Direct（网卡直读直写显存，全程不过 CPU——下一节专门拆解）。也能看出 MLA 模型天生适合 PD 分离——KV 比传统 GQA 小 4 倍多，同样网络能扛 4 倍并发。
三种搬法 ① P→D 直传（RDMA/GDR）：最常见，SGLang 默认走这条（Mooncake TransferEngine / NIXL 做搬运工）。 ② 共享 KV 池：P 写进全局多层存储（Mooncake Store：DRAM→NVMe），D 按需拉——P/D 数量可独立伸缩，前缀还能跨请求复用，代价是多一跳。 ③ 同节点 NVLink：单机把 P/D 分到不同卡组，教学/实验用，违背"独立伸缩"的本意。
05插播：RDMA 到底是什么？EFA 又是什么？ 可交互

上一节反复说"必须 RDMA"。RDMA（Remote Direct Memory Access，远程直接内存访问）是一种网络传输方式：网卡自己动手，直接读写对方机器的内存（乃至显存），CPU 和操作系统全程不插手。普通 TCP 则相反——每个包都要操作系统内核层层转手。按 ▶ 对比同一块 KV 走三条路径要倒几次手：

RDMA 是"能力"，不是某一种网卡——三个实现流派

InfiniBand（IB）：RDMA 的原生血统，专用网卡 + 专用交换机，NVIDIA 系数据中心标配（DeepSeek、Mooncake 的部署都是它）。RoCE：把 RDMA 协议跑在普通以太网上的开放标准，省掉专用交换机。AWS EFA：AWS 自研的第三条路——见下。SGLang 的 --disaggregation-ib-device mlx5_0 填的就是 IB/RoCE 网卡设备名。

那 EFA 是什么？——AWS 版的"类 RDMA"网卡

EFA（Elastic Fabric Adapter）是 AWS 给 HPC / 分布式 ML 实例（p4/p5/p6 等）配的高性能网卡。AWS 数据中心里没有 InfiniBand，EFA 就是它的替代答案：同样做到内核旁路 + 类 RDMA 语义 + GPU Direct，但底层协议是 AWS 自研的 SRD（对多路径友好的可靠传输），跑在以太网上。

对比普通 EC2 网卡（ENA，走 TCP/IP 内核栈）：延迟 ~30–50μs → ~15μs，带宽上到 400 Gbps（p5en）甚至 3.2 Tbps（p6）。软件不用为 EFA 单独写代码——它暴露标准的 libfabric 接口，NCCL 经 aws-ofi-nccl 插件用它，Mooncake / NIXL 也能把它当传输后端。

⚠️ 但 EFA 不是 InfiniBand：它提供的是"类 RDMA"接口，个别 IB 特性（如 GPU 直接发起通信的 IBGDA）不支持。在 AWS 上做 PD 分离，KV 传输走的就是 EFA 这条路。

和 NVLink 别搞混：一个管机内，一个管机外 NVLink 是同一台机器内 GPU↔GPU 的互联（H200 上 900 GB/s）；RDMA/EFA 是跨机器的网络（400 Gbps ≈ 50 GB/s，差一个数量级）。所以并行策略才有"机内 NVLink 随便通信（TP）、跨机要精打细算"的铁律——PD 分离把 KV 传输放到跨机链路上，正因为它是一次性搬运（每请求传一次），而不是 TP 那种每层每步都要的通信，才付得起这个税。
06SGLang 怎么实现：一次握手 + 一条非阻塞流水线 可交互

SGLang 的 PD 分离（0.4 实验、0.5+ 转正）设计得很"SGLang"：还是那个 launch_server，加一个 --disaggregation-mode 就切换人格；Router 把请求同时发给 P 和 D，两边拿同一个 bootstrap_room 编号对暗号；KV 用 RDMA 直写对方显存；两侧全部用非阻塞队列把传输藏进计算——和上一页 overlap scheduler 的哲学一脉相承：能藏的开销都藏起来。用 ≪ ≫ 逐步走一遍：

sglang_router --pd-disaggregation room #42
Prefill 实例
--disaggregation-mode prefill · 请求在 4 个队列间流转
① Bootstrap 队列等 D 端握手：拿到"KV 该写到哪"的显存地址
② Waiting 队列握手完成，等着被组进 prefill batch
③ GPU · Prefill forward算整段 KV + 首 token（radix cache 照常复用前缀）
④ Inflight 队列后台线程 RDMA 搬 KV，主循环不等、继续算下一批
Decode 实例
--disaggregation-mode decode · 镜像的接收流水线
① Prealloc 队列先把 KV 显存页分配好，把地址告诉 P（握手）
② Transfer 队列轮询传输状态——KV 正被 P 直写进来
③ Waiting 队列KV 到齐，等进 running batch
④ GPU · Decode 循环和其他请求 continuous batching，逐 token 流式输出

五个实现要点，对应上面五步

① 双发（不是接力）：Router 把请求同时发给 P 和 D 各一份，附带同一个 bootstrap_room ID。之后 P、D 之间的所有握手、传输都拿这个房间号配对——不需要中心化的状态服务。

② D 先分配、P 后写入：握手时 D 端先把 KV 显存页分配好，把页地址通过 P 端内嵌的 bootstrap server（挂在 tokenizer_manager 里的小 HTTP 服务）告诉 P。这样 P 算完 KV 就能单边 RDMA WRITE 直接写进 D 的显存——D 的 CPU/GPU 全程不用参与接收，轮询个完成标志就行。

③ 传输后端可插拔：--disaggregation-transfer-backend 选 mooncake（默认，Mooncake TransferEngine）或 nixl（NVIDIA 的传输库，Dynamo 同款），底下都是 GPU Direct RDMA，走 IB / RoCE / EFA。

④ 两侧都不阻塞：P 把传输扔给后台线程池（每 TP rank 4–12 条线程、多条并行传输队列）就去算下一个 batch；D 在等 KV 期间照常跑手头的 decode。传输完全藏在计算后面——这是 SGLang PD 分离吞吐不掉的关键。

⑤ P/D 配置解耦：P 池和 D 池的 --tp / --ep / --dp-size 可以完全不同（比如 P 用 TP16+EP16 吃算力、D 用 TP8+EP8 堆 batch）——这正是 PD 分离的核心收益。框架负责把不同切法的 KV 布局对上。

容错：分布式系统的税 拆成两池后，"半个请求"的失败模式出现了：D 等不到 KV（P 挂了）、P 传不出去（D 挂了）。SGLang 的对策是心跳 + 超时：D 每 5 秒探测 P 的 bootstrap server，连挂 2 次就把该 P 标记下线；握手和等 KV 各有超时（默认 300s，SGLANG_DISAGGREGATION_BOOTSTRAP_TIMEOUT / WAITING_TIMEOUT），超时的请求报错放弃、由上层重试。Router 侧多 P 多 D 实例互为备份。

08什么时候该上 PD 分离

PD 分离和 chunked prefill 解决的是同一个问题（prefill 干扰 decode），代价差一个数量级。先看对比，再对照经验法则：

维度Chunked Prefill（时间切片）PD 分离（空间隔离）
部署复杂度低——一个引擎参数高——多集群 + KV 传输 + Router
硬件门槛单节点就能用至少 2 套池 + RDMA 网络
TTFT中等（chunk 仍占时间片）极强（P 池专攻 prefill）
TPOT 稳定性中等（干扰变碎但仍在）极强（D 池零干扰）
长 prompt 高并发仍有卡顿几乎无卡顿
适用规模单节点 ~ 几节点十几 ~ 几百节点

✓ 值得上

同时满足越多，收益越大

· QPS 高且平均 prompt > 4K（prefill 压力实打实）
· TPOT 稳定性写进 SLO（线上对话、代码补全）
· 集群 > 16 GPU、有 RDMA 网络（IB / RoCE / EFA）
· MLA / GQA 模型——KV 小，传输便宜

✕ 别上

chunked prefill 就够了

· 单节点或 < 8 GPU——拆开后每池太小，反而两头都弱
· 没有 RDMA——KV 走 TCP，TTFT 不降反升
· benchmark / 实验负载——工程成本收不回
· 老 MHA 大模型——KV 太胖，传输税重

本项目的落点 真实数据流页的实测是 2×A10G 单节点、无 RDMA——按上表每一条都不该上 PD，所以那套部署用的是合并模式 + chunked prefill。PD 分离是"几十张卡、线上高并发"量级的武器；等以后在 AWS 上组多节点 + EFA 的实验，再补一组 P/D 分池的实测 trace 对比。

PD 分离 = 承认 prefill 和 decode 是两种生物，干脆分开养。
P 池吃算力、D 池吃带宽，各自把硬件榨到极限；代价是 KV cache 要用 RDMA 搬家。SGLang 的实现把这件事做成了一条非阻塞流水线：双发请求、D 先备好显存、P 算完单边直写、两侧照常干活——传输藏进计算，一如既往。
