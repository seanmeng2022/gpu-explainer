GPU EXPLAINER · 并发容量实战

GPU 全跑满，为什么还排队？
一台 8×H200 的并发账本

这是一次真实排障的完整复盘：一台 p5e.48xlarge（8×H200，1128GB 显存）用 SGLang 部署 GLM 大模型 FP8，供 10+ 人日常编码 agent 使用——结果 6 个并发就顶满，第 7 个开始排队，P95 TTFT 高达 20 秒。本页沿着当时的排查路径走一遍：GPU 利用率 → 引擎指标 → 缓存统计 → 启动日志，最后落到一个结论：并发上限 = KV cache 池容量 ÷ 单请求上下文占用。前面的 SGLang 引擎页讲原理，这一页讲这些原理在生产上怎么"算账"。

附：可交互并发计算器
01现象与部署配置

硬件不弱、量化拉满（FP8 权重 + FP8 KV cache）、缓存也开了三级 HiCache——但用户的体感是"10 个人一起用就有人要等"。先看启动命令，每一行右边注明它在这次排障里扮演的角色：

docker run … sglang serve（节选关键参数）
直觉会骗人：这套配置"看上去"没毛病 量化已经是 FP8 权重 + FP8 KV（各省一半显存），HiCache 三级缓存也开了,mem-fraction 0.88 也是常规值。没有一个参数写错了——问题出在这些参数组合起来之后,容量的账没人算过:307200 的上下文上限 × 重上下文 agent 负载,意味着单个请求就能吃掉巨量 KV。往下看证据链。
02四步证据链：从"感觉卡"到"定位到字节"

排障的顺序很重要：每一步先排除一层,再往下钻。四个证据分别来自 GPU 监控、/metrics 接口、网关 token 统计、启动日志——凑齐后结论唯一。

证据 ① · GPU 监控
8 卡利用率全程贴顶
≈100% 持续
偶发掉到 20%~80% 的毛刺可忽略（调度间隙）。关键信息：请求确实都进到了引擎，机器是真的忙。
排除 → 网关/代理层限流。瓶颈在引擎内部。
证据 ② · /metrics 实时指标
5 个请求吃掉 76% 的 KV 池
token_usage 0.76 @ 5 req
num_running_reqs=5、num_queue_reqs=1——才 5 个在跑,池子已用 76%,第 6 个已在排队。平均每个请求占整池 ~15%。
锁定 → KV cache 容量瓶颈,不是算力调度问题。
证据 ③ · 网关 12h Token 统计
前缀缓存其实工作得极好
95.5% 命中率
输入 494.29M token,其中 472.23M 命中缓存。注意:/metrics 里的 cache_hit_rate gauge 读数是 0——口径不可信,要用累计量对账。输入:输出 ≈ 113:1,典型重上下文 agent 负载。
排除 → 缓存失效。也意味着这里已无优化空间。
证据 ④ · 启动日志
池子总共就这么大
618,496 token
max_total_num_tokens=618496,同时 max_running_requests=2048（远不是限制）、available_gpu_mem=6.10GB（每卡仅剩 6GB 余量）。
定论 → 618496 × 0.76 ÷ 5 ≈ 单请求 94K token,上限 6.5 并发。
高峰期驾驶舱快照（后来补的仪表盘） KV 池 78% · 运行 6 个 / 等待 10 个（排队的比在跑的还多）· TTFT P95 = 19.2s · ITL P95 = 33.7ms。最后一个数字是好消息:请求一旦进池子,decode 很流畅(~30 token/s)。用户的痛苦 100% 来自"进不去",而不是"进去了慢"——所以优化方向唯一:腾容量。
03618496 这个数是谁定的？——SGLang 的显存盘点

max_total_num_tokens 不是默认值,是 SGLang 启动时现场算出来的:一道减法（显存预算 − 权重 − 激活）加一道除法（÷ 每 token 的 KV 字节数）。示意如下（宽度为示意比例,权重/激活的精确值以实际日志为准）:

能把池子变大的参数

这次部署里还剩的杠杆

· --mem-fraction-static ↑ — 预算变大,池子线性变大（唯一可动,但每卡只剩 6.1GB,只能 0.88→0.90 小步走）
· 权重精度 — 已 FP8,拉满
· --kv-cache-dtype fp8_e4m3 — 已 FP8,每 token KV 减半,拉满

不影响池子大小的参数

容易误会的两个

· --context-length — 只影响调度准入和单请求上限,不改变池子容量
· --hicache-size 128 — 那是 host 内存（CPU RAM）里的二级缓存,不占显存,与 618496 无关

每 token 的 KV 有多大？ 每 token KV 字节数 = 层数 × 2（K 和 V）× KV 头数 × head_dim × 每元素字节数。FP8 每元素 1 字节,BF16 是 2 字节——这就是 KV 量化直接让池子翻倍的原因。TP=8 时这份 KV 切到 8 卡,但 max_total_num_tokens 统计的是全局总量。想精确验证:启动日志里 KV Cache is allocated. #tokens: …, K size: … GB, V size: … GB,拿 (K+V) 的 GB ÷ token 数即得。
04并发账本计算器 可交互

整次排障最后收敛成一个除法:并发上限 ≈ KV 池容量 ÷ 平均单请求上下文占用。拖两个滑块感受一下——为什么"降低人均上下文"是不花钱版的扩容,而调 mem-fraction 只是小修小补:

>_ 左滑块 = 用户的使用习惯（会话多长）,右滑块 = 服务端显存水位。下面的池子条按本案例实测值初始化:0.88 水位、人均 94K。
重度编码 agent 会话普遍 60~150K;勤 compact / 勤开新会话能压到 32~48K;一个 300K 满长请求 ≈ 半个池子。
按"0.88 时池子 618,496"线性外推(近似)。本案例每卡仅剩 6.1GB,实际提到 0.93 会 OOM——超过 0.91 请当作纸面数字看。
理论并发上限 6
KV 池容量 618K token
10 人团队需等待 4 人
并发上来之后,下一个瓶颈立刻换人 GPU 本来就贴着 100%。并发从 6 放到 15+,排队消失了,但 decode 算力是大家分的——每个人的 token/s 会摊薄。好在本案例负载是 113:1 的重输入轻输出、且 95% 的 prefill 被缓存吃掉,体感恶化有限。但要"并发和单人速度都不降",最终答案只有加机器。
05P95 TTFT = 20 秒意味着什么

TTFT（Time To First Token,首字延迟）是把容量问题翻译成用户体感的那座桥。P95=20s 的意思是:每 20 个请求就有 1 个,从发出到看到第一个字等了超过 20 秒。这 20 秒花在哪,可以精确拆开:

TTFT = 排队时间 + prefill 时间

排队等 KV 池腾位置 ≈ 19s+ prefill <1s
池子满 → 调度器不放行,请求在 waiting 队列干等 95.5% 前缀命中 → 94K 会话只需新算 ~4K token,H200×8 上亚秒级

所以这 20 秒几乎全是排队——它不是一个新问题,而是"6.5 并发上限"在用户侧的直接投影。反过来,ITL P95 = 33.7ms 证明进了池子体验就很好。

06优化清单（按性价比排序）

诊断收敛后,能做的事其实不多——量化拉满、缓存命中 95%、调度参数无瓶颈,没有哪个配置项能凭空翻倍 KV 池。按投入产出排:

压低人均上下文占用 6 → 13+ 并发 · 零成本
并发随人均占用线性提升:94K→48K 直接翻倍。手段全在客户端侧:更积极的 compact（上下文到 60K 就压缩,别等逼近上限）、勤开新会话（一个会话干一件事）、少用"整仓库读进上下文"的用法、网关对单请求输入做软限制（>150K 告警）。
--mem-fraction-static 0.88 → 0.90 6 → ~8 并发
每卡挤 3~4GB 给 KV 池。不能贪:日志显示每卡仅剩 6.1GB,提到 0.93 会 OOM。改完对比重启前后的 max_total_num_tokens,再用满长请求压测确认。
--context-length 307200 → 131072 防垄断 · 公平性
作用不是扩池子,是禁止单个请求吃掉半池 KV——300K 上限下,一个满长请求就能让所有人一起卡。降到 128K 后超长会话被迫在客户端 compact。前提:与业务确认没有真 >128K 的场景。
加第二台机器 根治 · 15~20+ 并发
要支撑 15~20 并发且单人速度不降,单机没有免费午餐。关键细节:负载均衡必须做会话亲和性路由（同一会话固定同一台）——随机打散会把 95.5% 的 prefix cache 命中率打没,prefill 压力暴涨,等于花两台机器的钱买不到两倍容量。
07方法论速查:下次再遇到"并发不够"
症状 / 读数指向
GPU 利用率低 + 用户在等请求没进引擎 → 查网关、nginx worker、客户端连接池
GPU 贴 100% + token_usage 高 + queue 堆积KV 容量瓶颈（本案例）→ 算账:池容量 ÷ 人均占用
running 卡在固定数字、token_usage 却不高被 max-running-requests 等并发参数卡住 → 显式放开
queue≈0 但并发一多人人变慢纯吞吐瓶颈 → 并行策略（EP/TBO）、投机解码、加副本
cache_hit_rate gauge 显示 0%先别信,用累计命中 token 对账;或看逐请求日志 #cached-token
TTFT 高、ITL 低痛在"进不去"（排队/容量）,不在"算得慢"
TTFT 低、ITL 高decode 算力不足或 CUDA graph 未覆盖大 batch → cuda-graph-max-bs
有人发长请求时全员卡顿chunked prefill 配置、context-length 上限过大(单请求垄断)

LLM 服务的并发上限,本质是一道除法:KV 池容量 ÷ 人均上下文占用。
GPU 100% 不代表算力瓶颈——先看 token_usage 和 waiting 队列;缓存命中率要用累计量对账,别信瞬时 gauge; TTFT 高 + ITL 低,痛的是"进不去"而不是"算得慢"。
配置能挤的都是零头,把分母压小（compact）或把机器加倍,才是数量级的解。
