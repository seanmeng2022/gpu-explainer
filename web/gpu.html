§01CPU 与 GPU:两种设计哲学

同样的晶体管预算,CPU 花在让单个任务尽快完成(大缓存、分支预测、乱序执行),GPU 花在同时做尽量多份同样的活(把控制逻辑砍到最少,剩下全堆计算单元)。LLM 推理的主体是矩阵乘——几百万个"乘加"彼此独立,正中 GPU 下怀。

§02拆开一块 H100

GPU 的基本组织单位是 SM(Streaming Multiprocessor,流式多处理器)。H100 有 132 个 SM,每个 SM 是一台小型的向量计算机:自带计算单元、寄存器和一小块超快的共享内存。CUDA kernel 启动时,成千上万个线程块被撒到这些 SM 上并行执行。

H100 die · 132 个 SM 悬停任意小格

一次矩阵乘会被切成许多 tile,每个 tile 交给一个线程块,由某个 SM 执行。SM 越多、喂得越饱,吞吐越高——"GPU 利用率"本质上是"SM 有没有活干"。
128 × CUDA Core标量/向量运算,FP32·FP64·INT32
4 × Tensor Core专吃小矩阵乘(GEMM),BF16/FP8 下一拍完成一整块 MMA——LLM 算力主力,占 H100 峰值算力 ~90%
寄存器堆 256 KB每线程私有,零延迟
共享内存/L1 256 KB线程块内交换数据,手动管理的"草稿纸"
线程以 32 个一组(warp)锁步执行同一条指令;一个 SM 同时驻留几十个 warp,某个 warp 等显存时立刻切换到别的 warp——用并行掩盖延迟,这是 GPU 不需要大缓存的原因。

显存层级:每一层差一个数量级

算得快的前提是喂得上数据。从寄存器到 HBM,容量每大一级,带宽/延迟就差一个量级——高性能 kernel 的全部技巧,就是让数据尽量停留在上层(FlashAttention 的本质就是把注意力计算重排到共享内存里做完)。

Decode 阶段每生成 1 个 token 都要把全部权重从 HBM 读一遍,算力却只用一点——所以 decode 是显存带宽瓶颈(memory-bound),这是后面 PD 分离页的出发点。

§03单机多卡:PCIe → NVLink → NVSwitch

单卡 80 GB 放不下几百 GB 的模型权重,于是切给多卡(TP/EP)。但张量并行每一层都要在卡间同步一次——通信走什么管道,直接决定多卡是加速还是"等通信"。

默认管道:PCIe——所有设备的公共走廊

没有专用互联时,GPU 之间只能走 PCIe:数据先从 GPU-0 出发,经过 PCIe 交换机(甚至绕道 CPU 内存),再进 GPU-1。PCIe 5.0 ×16 单向 64 GB/s——听着不小,但和 HBM 的 3350 GB/s 比,差了 50 倍,而且这条走廊还要和网卡、SSD 共享。

NVIDIA 的答案是给 GPU 修专线:NVLink——点对点直连,不经过 CPU,不挤公共走廊。H100 每卡 18 条 NVLink 4 链路,合计双向 900 GB/s,是 PCIe 的 7 倍。

同一份 16 GB 权重分片,从 GPU-0 搬到 GPU-1:先走 PCIe(绕道 CPU),再走 NVLink 直连。注意路径和速度差。

NVLink 代际:每代翻倍

Blackwell 一代的 NVL72 机柜用 NVLink 5 + NVSwitch 把 72 块 GPU 连成一个统一高速域——"单机"的边界从 8 卡扩大到了一整个机柜。

NVSwitch:从"专线"到"全连接交换机"

NVLink 直连有个几何问题:8 块卡两两互连需要 28 条通路,每卡的 18 条链路摊薄后点对点带宽就低了。NVSwitch 是 NVLink 的交换芯片:每块 GPU 把全部 18 条链路接进交换平面(DGX H100 用 4 颗 NVSwitch3),于是任意两卡之间都是满速 900 GB/s,且互不争抢——all-to-all 流量(MoE 的 EP 通信)尤其受益。

DGX H100 · 8 卡全互联:先看单卡对单卡满速,再看 8 卡同时 all-to-all——交换平面下没有"绕路",也没有"堵车"。

§04多机多卡:跨节点,进入网络的世界

8 卡装不下 DeepSeek-V3(671B),训练更要几千卡——必须跨机器。NVLink 出不了机箱(NVL72 机柜除外),跨节点靠网卡 + 交换机:带宽再降一个台阶,延迟从百纳秒涨到微秒级。

InfiniBand
HPC 血统的专用网络,原生 RDMA、微秒级延迟、无损传输。NDR 一代每端口 400 Gb/s = 50 GB/s。大规模训练集群的主流选择。
DGX H100 每台配 8 张 ConnectX-7,每卡一张网卡
RoCE
RDMA over Converged Ethernet——在以太网上跑 RDMA。便宜、生态通用,但要精细调优(PFC/ECN 防丢包)才能接近 IB 的表现。
云厂商自建集群常用,省钱换运维复杂度
AWS EFA
AWS 自研的第三条路——不是 IB 也不是 RoCE。底层跑自研 SRD 协议:允许乱序到达、把包喷洒到多条路径,天生适配云的多路径网络,不需要 PFC 无损配置。OS-bypass + 支持 GPUDirect RDMA。p5en(8×H200)每实例 16 张 EFA,聚合 3.2 Tb/s = 400 GB/s,摊到每 GPU 与 IB NDR 同量级。
AWS 上跑多机训练/推理的唯一 RDMA 选择(不提供 IB)· NCCL 经 aws-ofi-nccl 插件走 libfabric;但 EFA 没有 IBGDA,不支持 GPU 直接发起 RDMA——这个坑由 UCCL-EP 来填(见 §05)
GPUDirect RDMA
让网卡直接读写 GPU 显存,数据不再经过 CPU 内存中转。跨机 GPU→GPU 延迟和 CPU 占用双降。IB / RoCE / EFA 三种网络都支持。
NCCL 检测到支持时自动启用

Rail-optimized 拓扑:8 张网卡,8 条"轨道"

§05集合通信:NCCL 的语言

应用层不直接操心 NVLink 还是 IB——大家调 NCCL(NVIDIA Collective Communications Library)。它把"多卡协同"抽象成几个集合通信原语,并自动探测拓扑、选最优路径(机内走 NVSwitch,跨机走 RDMA)和算法(ring / tree)。

AllReduce
每张卡各有一份数据,求和后每张卡都拿到完整结果。出现频率最高的原语。
用在 TP 每层的部分和聚合 · DP 梯度同步
AllGather
每张卡各持一片,结束后每张卡都集齐所有分片。
用在 TP 拼接激活 · ZeRO 取回参数分片
ReduceScatter
先求和、再把结果切片分给各卡——AllReduce 的前半程。
用在 ZeRO/FSDP 梯度分片
All-to-All
每张卡给每张卡发一份不同的数据,N² 条流同时跑——最吃交换带宽。
用在 MoE 专家并行(EP)的 token 分发/回收 → 见 MoE 页
Broadcast
一张卡的数据复制到所有卡。
用在 权重加载 · 采样参数下发
P2P Send/Recv
点对点收发,不是集合操作但同样走 NCCL。
用在 PP 级间传激活 · PD 分离传 KV cache → 见 PD 页
Ring AllReduce,NCCL 的看家算法:4 卡连成环,数据切 4 块。前半程 Reduce-Scatter(边传边加,3 步后每卡握有 1 块完整和),后半程 All-Gather(把完整块转 3 圈,人手一份)。柱高 = 该块已累加的份数,白框 = 完整。
 
为什么用环?每卡每步只和邻居通信,链路利用率 100%。每卡总通信量 = 2×(N−1)/N × 数据量 ≈ 2×数据量,与卡数 N 几乎无关——这就是 AllReduce 能扩展到几千卡的原因。代价是 2(N−1) 步的延迟,所以小消息 NCCL 会自动切换 tree 算法。

NCCL 之外:为 MoE / PD 分离定制的通信库

NCCL 的集合原语是"整齐"的——所有卡同步参与、数据量对称。但 MoE 的 all-to-all 每个 token 去向不同、数据量不对称,PD 分离的 KV 传输是异步点对点大块搬运——通用 NCCL 都不是最优解,于是出现了一批专用库:

DeepEP DeepSeek · 2025 开源
专为 EP 的 dispatch/combine 写的 all-to-all 内核。两大杀招:① GPU 直接发起 RDMA(NVSHMEM/IBGDA,连 CPU 都不惊动,token 级细粒度收发);② 两级转发——token 先走 IB 到目标节点上"同号 GPU",再走 NVLink 转给目标专家,两条管道同时打满,仅需 ~20 个 SM。低延迟模式用 hook 方式做通信-计算重叠,不占 SM。
DeepSeek-V3/R1 大 EP 推理的基石 · SGLang --enable-deepep-moe · 但绑死 NVIDIA GPU + ConnectX/IB
UCCL-EP UC Berkeley 等 · 2025
解决 DeepEP 的移植性:EFA、Broadcom 网卡没有 IBGDA,GPU 没法直接指挥网卡。UCCL-EP 把"发起"和"执行"拆开——GPU 照旧做 token 级路由决策,通过无锁 FIFO 把命令递给 CPU 代理线程,由 CPU 代发 GPUDirect RDMA。API 与 DeepEP 兼容,支持 NVIDIA/AMD GPU × ConnectX/EFA/Broadcom 网卡。
在 EFA 上比现有方案快至 2.1×,SGLang token 吞吐 +40% ——在 AWS 上跑大 EP 就靠它(本项目的场景!)
Mooncake Moonshot/Kimi · FAST'25 最佳论文
不是 EP 库,而是 KV cache 为中心的一整套 PD 分离架构:Transfer Engine 负责跨节点零拷贝搬 KV(RDMA 多网卡聚合、拓扑感知);Mooncake Store 把集群里闲置的 CPU 内存 + SSD 组成分布式 KV 池,prefill 结果全局复用。
Kimi 生产系统,长上下文吞吐最高 +525% · SGLang/vLLM 的 PD 分离传输后端之一 → 见 PD 分离页

一句话分工:NCCL 管整齐的集合通信(TP/DP),DeepEP / UCCL-EP 管 MoE 的不规则 all-to-all(UCCL-EP = 能跑在 EFA/AMD 上的 DeepEP),Mooncake 管 PD 分离的 KV cache 搬运和存储。

§06带宽全景:一张梯子记住所有数字

把本页出现过的管道排在一起(对数刻度)。每下一级台阶,慢一个数量级——系统设计的所有取舍都写在这张梯子上。

读法:TP 的每层 AllReduce 必须留在 NVLink 层(所以 TP 一般不出单机);EP 的 all-to-all 尽量留在 NVSwitch 域;跨机只放低频大粒度通信(PP 传激活、DP 同步梯度、PD 分离传 KV);而 decode 的根本瓶颈 HBM 带宽,在梯子最顶端——这条排序,决定了后面每一页的架构选择。

带走这 5 句话

  1. GPU 赢在吞吐:上万个简单核心 + SIMT + 高带宽显存,专为"海量同构计算"(矩阵乘)设计。
  2. SM 是执行单位,Tensor Core 是算力主力;显存层级每层差一个数量级,快 kernel 的秘诀是少碰 HBM。
  3. 单机多卡:PCIe 是公共走廊(64 GB/s),NVLink 是专线(900 GB/s),NVSwitch 把 8 卡连成无争抢的全交换域。
  4. 多机多卡:InfiniBand/RoCE(自建)或 EFA(AWS)+ GPUDirect RDMA,网卡直读显存;rail 拓扑让 GPU-i ↔ GPU-i 一跳直达。
  5. NCCL 原语是通信的词汇表:TP→AllReduce(机内),EP→All-to-All,PP/PD→P2P(可跨机);不规则通信有专用库——MoE 用 DeepEP(IB)/UCCL-EP(EFA),PD 分离的 KV 搬运用 Mooncake。
