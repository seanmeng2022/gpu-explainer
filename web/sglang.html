GPU EXPLAINER · SGLANG 引擎

模型只管算一层 forward，
SGLang 管把请求喂得又快又满

前面几页讲的是模型本身怎么算（注意力、MoE）。但真把一个模型变成能扛住成千上万并发的线上服务，瓶颈往往不在 GPU 算得多快，而在 CPU 怎么调度、KV cache 怎么复用、batch 怎么组。SGLang 就是干这个的推理服务引擎——它的三板斧是 RadixAttention（前缀复用）、零开销调度器（CPU 不拖 GPU 后腿）、continuous batching（持续组 batch）。本页把这三件事拆开看。

01SGLang 在整个栈里的位置

一个推理引擎（SGLang / vLLM / TGI）夹在用户请求和裸 GPU kernel 之间。它不改模型的数学，只负责：把请求排进 batch、管显存、复用缓存、把活塞满 GPU。SGLang 的特色是后端运行时 + 前端语言协同设计。

一句话区分"模型"和"引擎" 模型（DeepSeek、Llama）决定"一次 forward 算什么"；引擎（SGLang）决定"什么时候、为谁、和谁拼在一起算这次 forward"。同一个模型，换个引擎，吞吐能差好几倍——差的全是 CPU 侧的调度与缓存功夫。本页讲的就是这部分。
02RadixAttention：别重算别人算过的前缀 可交互

真实负载里大量请求共享前缀：同一个 system prompt、多轮对话的历史、RAG 里同一段文档、agent 里同一套工具定义。朴素引擎对每个请求把整段 prompt 的 KV 从头算一遍——纯属浪费。SGLang 把所有请求的 token 序列存进一棵 radix tree（基数树），相同前缀的 KV cache 只算一次、之后全部命中复用。点下面的请求，看哪些前缀是缓存命中（绿）、哪些是新算（黄）。

为什么用"基数树"，而不是普通的哈希缓存？

普通缓存只能整段命中（key 一样才算 hit）。而 prompt 是逐 token 共享的——"你是一个助手。"和"你是一个助手。请翻译"共享前 6 个字。radix tree 把公共前缀压成同一条路径，新请求沿树往下走，能匹配多深就复用多深，剩下的尾巴才去算 prefill。这也是为什么 prefix 命中率对吞吐影响巨大——命中得越深，省掉的 prefill 越多。

注意：radix tree 节点上挂的是 KV cache 的物理页指针，不是文本。缓存满了按 LRU 淘汰叶子节点。SGLang 后来还把它扩成多层 HiCache（GPU → CPU 内存 → 外部存储三级），冷前缀下沉、热前缀留在显存。

命中率会"塌方"——前缀顺序必须一致 radix tree 是严格前缀匹配：只要开头差一个 token（比如把 system prompt 和用户内容拼接顺序换了、或时间戳混进了前缀），整条路径就匹配不上，退化成全量 prefill。所以工程上要把固定不变的内容放最前面、动态内容放后面，才能稳定吃到缓存。
03零开销调度器：别让 CPU 把 GPU 晾在那 可交互

每跑一步 forward，CPU 都得先干一堆活：决定这步组哪些请求、分配 KV 显存、做 radix 前缀匹配、准备 kernel 参数。朴素做法是串行的——CPU 准备完，GPU 才开算；GPU 算完，CPU 再准备下一步。一个没优化的引擎能把近一半时间花在 CPU 上，GPU 干等。SGLang 的解法：让 CPU 提前一个 batch 干活，和 GPU 计算重叠起来（overlap scheduler，v0.4 起默认开启）。按 ▶ 播放，同一根播放头从左到右扫过两种模式，块随扫过逐步"长"出来：

怎么做到的：调度器"跑在 GPU 前面一步"

关键 trick：CPU 不等 GPU 这一步的结果，就先去准备下一步的 batch 元数据。可第 N+1 步的输入不正是第 N 步刚生成的 token 吗？SGLang 用 future token（占位的未来 token）解依赖：先用占位符把下一步的 batch 排好、显存分好，等 GPU 真吐出第 N 步的 token，再把占位符替换成真值。配合精心安排的 CUDA event 同步，GPU 就能一步接一步永不停转。官方实测约 +1.1× 吞吐。

这思路和模型本身的 CUDA graph、continuous batching 是配套的：CUDA graph 把每步的 kernel 序列录下来重放、砍掉逐 kernel 的 launch 开销；overlap scheduler 把"组下一个 batch"这件 CPU 活藏到 GPU 算的时间里。两个一起，CPU 才彻底不拖后腿。

04Continuous batching：每步都重新组队 可交互

调度器调度的"batch"不是训练里那种"凑够 N 个一起算"的静态批。线上请求随时来、随时完成、长度还都不一样。SGLang 维护一个 running batch，每生成一个 token 就重新决定谁进谁出——完成的请求立刻让位，新来的请求填进空槽。下面 4 个 GPU 槽位、上下两种策略跑同一批请求，盯着右边的利用率数字看：

✓ Continuous batching

每个 token 后重新组 batch

A 生成完立刻移出、空槽马上塞进新来的 D。GPU 始终满载，新请求几乎零排队。这是现代推理引擎的标配。

和 Prefill / Decode 两阶段的关系 新到的请求要先 Prefill（一次性算完整段输入的 KV，算力重），之后才进入逐 token 的 Decode（复用 KV，访存重）。这两类活的特性不同，调度器要决定怎么拼——比如 chunked prefill 把长 prefill 切片，插进 decode 步里，避免一个大 prefill 把所有人的 decode 卡住。→ 在真实数据流页面看 Prefill/Decode 的实测 trace，那里的步骤正是 SGLang 内置 torch.profiler 抓出来的。
05多卡并行：TP / DP / EP 怎么选 可交互

DeepSeek、Mixtral 这类 MoE 模型动辄几百 GB，单卡装不下，得切到多张卡。关键认知：TP/DP/EP 不是三选一，而是给模型的不同部分分别选、组合起来用——attention 层走 TP 或 DP，MoE 专家层走 EP。这也是真实配置里 --tp 8 --dp-size 8 --ep 8 能同时出现的原因。

TP · 张量并行
Tensor Parallelism
把每一层的权重矩阵按行/列切到多张卡，每张卡算一片，再 all-reduce 合起来。每步都通信，最好走 NVLink。单卡装不下 / 低并发求低延迟时用。
DP · 数据并行
Data Parallelism
把整个模型复制多份（每份都有完整参数），每份独立处理不同批请求，副本间几乎不通信。高并发求吞吐时用，吞吐近线性扩展（配 SGLang Router 分流）。注意：和下面「问题一」的 DP Attention（DPA）不是一回事，见辨析。
EP · 专家并行
Expert Parallelism
MoE 专属：把不同的专家放到不同卡上。token 路由到哪个专家，就把它的激活值 all-to-all 发到那张卡，算完再发回。专家分散，显存压力小，但通信量大。
PD · 分离
Prefill / Decode Disaggregation
把算力重的 Prefill 和访存重的 Decode 放到不同的 GPU 池子，各自用最优配置，互不干扰。SGLang 在 96 张 H100 上演示过 PD 分离 + 大规模 EP。
通信重叠
TBO / SBO Overlap
EP 的 all-to-all 通信很贵。SGLang 用 双批重叠（TBO）把通信藏到另一批的计算后面，--enable-two-batch-overlap 最多再 ~2× 吞吐。和 §03 的思路一脉相承：能藏的开销都藏起来。
本项目的实测落点 实测跑的是 DeepSeek-V2-Lite, TP=2, 2×A10G：每卡权重 14.5GB（MoE 专家占 95%），SM 活跃 ~62%、HBM 活跃 ~58%。A10G 没有 NVLink，TP 通信走 PCIe——这正是为什么生产 MoE 想要 NVLink/H200 的直观原因。→ 真实数据流页面有逐卡权重分布和 DCGM 利用率读数。

怎么选？官方指导归纳成两个问题

SGLang 官方文档与大规模 EP 博客的建议，归结为先问两件事：① 模型的注意力是不是 MLA？② 你的负载是低并发求延迟、还是高并发求吞吐？

问题一：是不是 MLA 模型？——决定 attention 走 TP 还是 DP

先理清一件最容易绕晕的事：TP 和 DP 切的根本不是同一个东西。多卡并行能分两种"活"——① 一个请求内部的计算，② 一批不同的请求。TP 切①，DP 切②。

TP「按 head 切」
多张卡合力算 · 同 · 一 · 个请求
把请求 A 这一层的计算横向拆开摊给 4 卡：
卡0A·head1-4
卡1A·head5-8
卡2A·head9-12
卡3A·head13-16
4 卡只算出 A 的碎片，每层要 all-reduce 拼回 → 卡间频繁通信。单请求延迟低，但 MLA 只有 1 个 head，这种拆法拆不动。
每层 all-reduce 通信次数：0
DPA「按请求切」
一张卡独 · 包 · 几个完整请求
不拆单个请求，整个请求塞给一张卡，不同请求去不同卡：
卡0请求A请求B
卡1请求C请求D
卡2请求E请求F
卡3请求G请求H
每卡独立算完自己那几个请求，各存各的 KV。请求间天生独立 → 不用 all-reduce、不复制 KV。高并发吞吐高。
attention 卡间通信次数：0（恒为零）
一个餐厅的比喻（展开）
来了 8 桌客人要出菜。TP（按 head 切）= 一道菜由多个厨师流水线协作：切配、爆炒、摆盘各管一段，做一道菜要不停传递（=通信），单道菜出得快。DPA（按请求切）= 一个厨师全包几桌：1 号厨师独立做完第 1、2 桌的所有菜，2 号包第 3、4 桌……厨师互不打扰（=不通信）、各管各的食材（=各存各的 KV），8 人火力全开总出菜量大。"按请求切"就是后者：不拆单个请求的活，而是把整桌整桌的请求分给不同的卡。

理解了"切的是不同东西"，再看 KV cache 在两种切法下分别怎么放。每个 token 会缓存 K/V 向量，注意力是多头的、每头一套独立 K/V——所以 KV cache 像一排抽屉，一个头一个抽屉，"KV head 数"就是抽屉数。TP 省显存正是按抽屉（head）分卡：

① 普通模型（MHA/GQA）
抽屉多 → TP 切得动 ✓
按 head 把抽屉分给各卡
卡0头1头2
卡1头3头4
…头…
各卡只存自己那几个抽屉，KV 显存真正 ÷N。
② MLA + TP
只有 1 个抽屉 → 切不动 ✗
MLA 把所有头压成 1 份共享 KV
卡0整份KV
卡1整份KV
…整份KV
1 个抽屉分不给 8 卡，只能每卡复制一整份——非但没省，占了 N 份 KV。
③ MLA + DPA
换方向：按请求切 ✓
不按抽屉，按"请求"分卡
卡0请求A请求B
卡1请求C请求D
……
各卡专管不同请求、各存各的 KV，互不复制。不同请求注意力本就独立，天然不用卡间通信。

一句话：TP 想按 head 切 KV，可 MLA（DeepSeek V2/V3/R1）只有 1 个 KV head，切不动、只能复制，显存全浪费。SGLang 的 DP Attention（DPA） 改成按请求切，绕开这个问题——用 --enable-dp-attention 开启（要求 --dp-size > 1 且 tp_size % dp_size == 0）。

MLA 的标配是 DPA；普通 GQA（Llama / Qwen）抽屉够分，常规 TP 就行。所以"MoE + MLA" 的典型配置 = attention 走 DPA、专家走 EP，即 DeepSeek 官方推荐的 --tp 8 --dp-size 8 --ep 8 --enable-dp-attention。

别被名字带偏：DPA ≠ 上面卡片里的 DP（数据并行） 两者都叫 "DP"，但复制的范围完全不同：
· DP（数据并行）：整个模型复制 N 份，每份都有完整参数，各自是独立引擎，靠 Router 在引擎之间分流请求。显存代价 = N 份完整模型。
· DPA（DP Attention）：发生在一组卡（一个 TP 组）内部，且只针对 attention 这一段——这组卡上其余部分（MoE 专家走 EP、embedding/LM head 走 TP）依然是切开的，谁也没有完整模型。之所以带 "DP" 三个字母，只是因为它对 attention 用了"按请求分工"这个数据并行的思想：既然 MLA 的 KV 没有 head 维度可切，那干脆每张卡独立算自己那几个请求的注意力、各存各的 KV。
判断口诀：DP 复制的是"整个模型"，DPA 复制的只是"attention 权重"（每卡一份 attention 权重 + 各自请求的 KV），模型主体（占大头的专家）仍然是切分的。所以 --dp-size 8 配合 --enable-dp-attention 时并不会把显存变成 8 份模型——它只是 attention 的切法说明。
等等——既然 attention 按请求切，为啥还写 --tp 8？（展开）
因为 --tp 8 --dp-size 8 --ep 8 不是 8×8×8 张卡，而是同一组 8 张卡的三种"切法说明"。--tp 8 主要声明总共 8 张卡（池子大小），--dp-size 和 --ep 再分别规定模型的不同部分在这 8 卡上怎么并行：
· attention 层 → 看 dp-size：tp=dp=8 时 attention 纯按请求切（零 head 切分）；若 dp=4 则组间按请求、组内 2 卡再按 head 切（混合）。所以真正决定 attention 切法的是 dp-size，不是 tp。约束 tp_size % dp_size == 0 说的就是"同一个池子怎么再分组"。
· MoE 专家层 → 看 ep：按专家切。
· 其余部分（embedding、LM head、前几层稠密 FFN）→ 仍按常规 TP 按权重切——这才是 tp 还要写满 8 的原因。
一句话：tp = 用几张卡；dp-size = attention 怎么切；ep = 专家怎么切。三者共享同一组卡，各管一部分。

问题二：低并发还是高并发？——决定 attention 走 TP 还是 DP（性能交叉点）

低并发 / 求低延迟（交互聊天）→ TP + EP：单个请求被切到多卡并行算，单请求延迟最低。
高并发 / 求高吞吐（批处理、API）→ DP + EP：多副本各扛一批，吞吐近线性扩展。
这俩存在一个性能交叉点：并发低时 TP 赢，并发高时 DP 赢——务必在你自己的真实负载上测，别只看合成 benchmark。

EP 开了之后还要管两件事 ① 通信重叠：EP 的 all-to-all 很贵，开 TBO（--enable-two-batch-overlap）把它藏进另一批计算，最多再 ~2×。② 负载均衡：专家路由天然不均，开 EPLB（--enable-eplb）按激活统计重排/复制专家，削峰填谷。纯 Dense 模型没有专家，EP 不适用，只在 TP / DP 之间选。
