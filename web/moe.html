MoE（混合专家）：把一个大 FFN 拆成 N 个小专家，每个 token 只让 Router 挑 Top-K 个处理——知识容量巨大，单 token 算力很小。本页拆开看：Router 怎么挑、为什么省、代价是什么。

01同一层 FFN，两种活法 动画

先回忆 Transformer 页讲的：一层 = Attention（混上下文）+ FFN（每个 token 独立加工、存知识）。MoE 动的只是 FFN 这一半——把它从"一个大的"换成"一堆小的 + 一个调度员"。

Dense · 全员加班

每个 token 跑完整个大 FFN

一个又宽又大的 FFN，所有 token、所有参数，每次全激活。简单稳定，但参数量直接等于算力成本——想更聪明就得更贵。

MoE · 按需调度

N 个小专家，每个 token 只用 Top-K 个

把大 FFN 拆成 N 个小 FFN（专家）+ 一个 Router。每个 token 只点亮少数几个专家，其余休眠。总参数大、单次激活小。

为什么是拆 FFN，不是拆 Attention？ 因为 FFN 占了一层约 2/3 的参数（Attention 的 Q/K/V/O 投影只占约 1/3）。模型的事实知识主要存在 FFN 里（见 Transformer 页的"档案墙"demo）。拆 FFN 才能在"省算力"和"保知识容量"上同时拿到收益；拆 Attention 收益小、还会破坏跨 token 的信息混合。
02Router：谁来决定 token 去哪几个专家 可交互

MoE 层的核心是个小小的 Router（门控网络）——一个线性层。它给当前 token 的向量，对 N 个专家各打一个分，选分最高的 Top-K 个，再把这几个专家的输出按分数加权求和。点下面任意一个词，看 Router 怎么为它挑专家：

专家的"专长"是训练涌现的，不是人指派的

这 N 个专家初始全是随机数、毫无区别。训练时 Router 和专家一起学：哪个专家恰好把某类 token 处理得让 loss 更低，梯度就会把这类 token 越来越多地路由给它，它也就越来越擅长这类——正反馈下自然分化出"代码专家""标点专家""某语言专家"等。和多头注意力的"头分工"是同一个道理：结构是人设的，分工是学出来的。

注意：上面 demo 给专家标了"代码/数学/中文"等标签只为直观。真实专家的分工往往没有这么干净的人类可读语义，是统计意义上的聚类。

共享专家（Shared Expert） DeepSeek 等模型还设一个所有 token 都必经的共享专家，专学"通用知识"（语法、常识），让 Top-K 路由专家能腾出来学更细分的本事。所以实际是：1 个共享专家（恒亮）+ Top-K 个路由专家（按需亮）。
03关键解耦：知识容量 ↔ 推理算力

MoE 最重要的一句话——把"模型有多少知识"和"每个 token 要算多少"拆开了。Dense 模型这俩是绑死的，MoE 把它们解耦：

容量 ∝ 总参数，算力 ∝ 激活参数

知识容量取决于总参数（所有专家加起来）；推理 FLOPs 只取决于每个 token 实际激活的参数（共享专家 + Top-K）。两者不再相等——这就是 MoE 能"又大又便宜"的根。

以 DeepSeek-V3 为例：

671B 总参数中，每个 token 只激活 671B —— 亮着的格子才在干活（256 路由专家格中仅 Top-8 + 1 共享）

知识接近 671B 的 Dense 模型，但每个 token 的算力成本只相当于 37B 的 Dense 模型。这正是 Scaling Law 的另一条路：在同样推理预算下，MoE 能装下数倍于 Dense 的知识。难怪 DeepSeek-V3、Mixtral、Qwen-MoE、（被广泛猜测的）GPT-4 都走这条路。

04天下没有免费午餐：显存 & 通信

FLOPs 省了，代价转移到了显存和通信上——这也是 MoE 工程复杂的根源，正是 SGLang 页讲的 --ep（专家并行）要解决的事。

① 显存压力：所有专家都得常驻 GPU

虽然每个 token 只用 Top-K，但你事先不知道哪个 token 会用哪个专家，所以 N 个专家必须全部装在显存里待命。这就是为什么 MoE 的显存 ≈ 总参数（而非激活参数）。671B 的 DeepSeek-V3 即便 FP8 也要约 700GB，单卡装不下——必须 Expert Parallelism（EP）：把不同专家分到不同 GPU。

② All-to-All 通信：token 要"飞"到专家所在的卡 动画

专家分散在不同 GPU 后，token 被路由到的专家可能在别的卡上。于是每个 MoE 层要做两次 All-to-All：把 token 的激活值发到目标专家的卡（下图第一拍），专家加工（第二拍），再发回来（第三拍）。点的颜色 = 它要去的目标卡：

每层两次 All-to-All，对带宽要求极高。SGLang 用 双批重叠（TBO）把这通信藏到另一批的计算后面。这也是真实数据流页面里能看到 NCCL 通信的原因。

③ 负载均衡：别让几个专家累死、其余闲死 可交互

如果 Router "偏心"，少数专家被挤爆、其余没人用——叫 专家坍塌（Expert Collapse）：整批的延迟由最忙的专家决定，热门专家排长队时，闲专家再多也帮不上。切换下面两种 Router 感受差别：

整批延迟(由最忙专家决定) 1.0×

传统解法是加辅助损失惩罚不均衡；DeepSeek-V3 用 Auxiliary-Loss-Free 策略——给每个专家维护一个 bias 动态调节路由偏好，无需辅助损失也能均衡。

05另一条省钱线：MLA 压薄注意力的 KV
先说清归属：MLA 不属于 MoE，是正交的另一招 一个 Transformer 层 = Attention + FFN。MoE 优化的是 FFN 那一半（拆成专家、省算力）；MLA 优化的是 Attention 那一半（压 KV cache、省显存与带宽）。两者是两条正交的省钱线，只是 DeepSeek 同时用了，容易让人以为 MLA 是 MoE 的一部分——其实不是。放在这里，是因为它和 MoE 一起回答了同一个问题：DeepSeek 凭什么又大又便宜。

MLA（Multi-head Latent Attention，多头潜在注意力）是 DeepSeek-V2 提出的省 KV cache 的注意力变体。痛点：decode 阶段 KV cache 又大又要反复读，是显存和带宽的头号负担。减小 KV cache 有一条演化线，MLA 是最新一站：

MLA 的核心 trick：缓存一个"压缩潜向量"，用时再还原

标准多头注意力（MHA）要缓存每个头各自的完整 K、V。MLA 不这么干——它把多头 K/V 联合压缩成一个低维"潜向量"（latent）缓存起来（每个 token 一份）；真正算注意力时，再用学好的升维矩阵把潜向量还原成各头的 K/V。缓存里存的是一份压缩小向量，不是每头一套 K/V——所以 KV cache 能小到 MHA 的约 1/4 甚至更低，质量还几乎不掉。

这也解释了一个反直觉点：从分片角度看，这份共享潜向量没有"头"这个维度可切，常被描述成"只有 1 个 KV head"。于是张量并行（TP）想按 head 切 KV 时切不动、只能每卡复制一份——SGLang 为此改用按请求切的 DP Attention（DPA）。→ SGLang 页 §05「问题一」用"抽屉"图示讲透了这件事。

06落到本项目的实测
真实 trace 跑的就是一个 MoE + MLA 模型 本项目实测 DeepSeek-V2-Lite, TP=2, 2×A10G：每卡权重 14.5GB，其中 MoE 专家占 95%——这就是 §04 说的"显存 ≈ 总参数、专家全常驻"的直接体现。它的注意力正是 MLA（§05）。SM 活跃 ~62%、HBM 活跃 ~58%。A10G 没有 NVLink，EP/TP 通信走 PCIe，所以 All-to-All 更容易成为瓶颈。→ 在真实数据流页面看逐卡权重分布、PCIe/NCCL 实测读数。
07Dense vs MoE 速查

什么时候选哪个？ 小模型(<10B)、单卡/少卡、追求训练稳定 → Dense（MoE 的 overhead 划不来）。大规模预训练、有多卡 NVLink/IB 集群、要极致知识容量 → MoE。

Dense 是"全员加班"，MoE 是"按需调度"。
MoE 把知识容量（∝ 总参数）和推理算力（∝ 激活参数）解耦——用显存常驻和 All-to-All 通信这两份工程代价，换来了"又大又便宜"。模型决定算什么，Router 决定让谁来算。
